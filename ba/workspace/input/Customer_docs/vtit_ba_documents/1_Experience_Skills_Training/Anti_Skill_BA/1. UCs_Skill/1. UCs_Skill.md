# UC & Transaction Writer cho BA

Skill này hỗ trợ Business Analyst viết **Use Case (UCs)** và **Transaction** theo hướng rõ ràng, đồng nhất, không trùng lặp và có thể dùng trực tiếp trong tài liệu phân tích nghiệp vụ.

## Mục tiêu cốt lõi

Khi được yêu cầu viết Use Case hoặc Transaction, hãy tạo nội dung:

- đúng phạm vi nghiệp vụ/hệ thống;
- đặt tên Use Case theo cấu trúc nhất quán;
- tách Transaction theo từng hành động/function cụ thể;
- mô tả rõ tác nhân, hành động, phản hồi của phần mềm/hệ thống;
- bao phủ đủ nhóm UI, tích hợp hệ thống và Trục tích hợp dữ liệu khi liên quan;
- tránh trùng Use Case và Transaction trong toàn bộ tài liệu;
- viết theo cách Dev hiểu, QA kiểm thử được, BA Lead dễ review.

## Nguyên tắc bắt buộc

1. Một màn hình có nhiều function thì **mỗi function là một Transaction riêng**.
2. Không viết Transaction quá chung chung như “người dùng quản lý dữ liệu”.
3. Không gộp nhiều hành động khác bản chất vào một Transaction nếu chúng có luồng kiểm tra/xử lý khác nhau.
4. Không được phép trùng tên Use Case hoặc trùng Transaction trong cùng tài liệu.
5. Luôn nêu rõ hệ thống/phần mềm nào gửi yêu cầu, hệ thống/phần mềm nào xử lý, hệ thống/phần mềm nào nhận phản hồi.
6. Khi thiếu thông tin, dùng placeholder rõ ràng như `[Tên nghiệp vụ]`, `[Tên dữ liệu]`, `[Tên hệ thống nguồn]`, `[Tên hệ thống đích]` thay vì tự đoán nghiệp vụ quan trọng.
7. Ưu tiên câu mô tả theo mẫu: **Tác nhân thực hiện hành động. Phần mềm/Hệ thống kiểm tra hoặc xử lý và phản hồi kết quả.**
8. Nếu có xác thực, kiểm tra hợp lệ, ghi log, phản hồi lỗi, hãy thể hiện ngay trong Transaction.
9. Với Transaction tích hợp, phải làm rõ chiều dữ liệu: gửi dữ liệu, nhận dữ liệu, khai thác dữ liệu, phản hồi, ghi nhật ký.
10. Với UI quản lý nghiệp vụ, phải rà đủ các function phổ biến: xem danh sách, thêm mới, tìm kiếm, lọc, cập nhật, xóa, xem chi tiết, khóa/mở khóa, tải xuống, xem lịch sử thao tác.

## Loại đầu vào có thể nhận

Có thể nhận một hoặc nhiều loại thông tin sau:

- tên phân hệ/module;
- tên màn hình;
- danh sách function trên màn hình;
- mô tả nghiệp vụ;
- tài liệu SRS/BRD/URD;
- mockup/wireframe;
- danh sách API hoặc luồng tích hợp;
- tên hệ thống nguồn/hệ thống đích;
- danh sách dữ liệu trao đổi;
- tài liệu Use Case hiện có cần review.

Khi đầu vào chưa đủ, vẫn tạo bản nháp tốt nhất và thêm mục **Giả định / Câu hỏi mở**.

## Quy trình xử lý mặc định

Khi viết UCs/Transaction, thực hiện theo thứ tự:

1. Xác định nhóm Use Case:
   - UI quản lý nghiệp vụ;
   - UI cấu hình/tích hợp hệ thống;
   - tích hợp giữa phân hệ với phân hệ;
   - tích hợp giữa phân hệ với hệ thống;
   - Trục tích hợp dữ liệu;
   - nhóm khác nếu người dùng cung cấp.
2. Xác định tác nhân chính:
   - người dùng nghiệp vụ;
   - quản trị hệ thống;
   - hệ thống nguồn;
   - hệ thống đích;
   - Phần mềm Trục tích hợp dữ liệu;
   - service/module liên quan.
3. Đặt tên Use Case.
4. Tách Transaction theo từng function/hành động.
5. Viết Transaction theo mẫu chuẩn.
6. Review trùng lặp và thiếu luồng.
7. Xuất kết quả dạng bảng.

## Mode A: Viết Use Case cho UI quản lý nghiệp vụ

Dùng khi màn hình là màn hình quản lý dữ liệu/nghiệp vụ, ví dụ: quản lý người dùng, quản lý hồ sơ, quản lý danh mục, quản lý cấu hình nghiệp vụ.

### Cấu trúc tên Use Case

```text
Quản lý [Tên nghiệp vụ/dữ liệu quản lý]
```

Ví dụ:

```text
Quản lý người dùng
Quản lý danh mục đơn vị
Quản lý hồ sơ yêu cầu
Quản lý cấu hình biểu mẫu
```

### Bộ Transaction chuẩn cho UI quản lý

Khi người dùng chỉ cung cấp tên nghiệp vụ/dữ liệu, hãy đề xuất bộ Transaction mặc định sau và điều chỉnh theo function thực tế:

| STT | Nhóm function | Mẫu Transaction |
|---:|---|---|
| 1 | Xem danh sách | `[Tác nhân] xem danh sách [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm hiển thị danh sách [Tên nghiệp vụ/dữ liệu quản lý].` |
| 2 | Khởi tạo/Thêm mới | `[Tác nhân] khởi tạo [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ thực hiện lưu thông tin và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| 3 | Tìm kiếm | `[Tác nhân] tìm kiếm danh sách [Tên nghiệp vụ/dữ liệu quản lý] theo tiêu chí. Phần mềm hiển thị danh sách [Tên nghiệp vụ/dữ liệu quản lý] theo kết quả tìm kiếm tương ứng.` |
| 4 | Lọc | `[Tác nhân] lọc danh sách [Tên nghiệp vụ/dữ liệu quản lý] theo tiêu chí chọn. Phần mềm hiển thị danh sách [Tên nghiệp vụ/dữ liệu quản lý] theo kết quả lọc tương ứng.` |
| 5 | Cập nhật | `[Tác nhân] cập nhật thông tin [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ xử lý yêu cầu cập nhật dữ liệu và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| 6 | Xóa | `[Tác nhân] xóa [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ xử lý yêu cầu cập nhật dữ liệu và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| 7 | Xem chi tiết | `[Tác nhân] xem chi tiết thông tin [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm hiển thị thông tin chi tiết của [Tên nghiệp vụ/dữ liệu quản lý].` |
| 8 | Khóa/Mở khóa | `[Tác nhân] khóa/mở khóa [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ cập nhật trạng thái khóa/mở khóa và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| 9 | Tải xuống/Xuất file | `[Tác nhân] tải xuống danh sách [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm tạo và tải xuống file danh sách [Tên nghiệp vụ/dữ liệu quản lý].` |
| 10 | Xem lịch sử thao tác | `[Tác nhân] xem lịch sử thao tác [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm hiển thị lịch sử thao tác của [Tên nghiệp vụ/dữ liệu quản lý] tương ứng.` |

### Checklist chống sót cho UI

Trước khi kết luận danh sách Transaction đã đủ, kiểm tra:

- Có function xem danh sách không?
- Có thêm mới/khởi tạo không?
- Có tìm kiếm không?
- Có lọc không?
- Có cập nhật không?
- Có xóa không?
- Có xem chi tiết không?
- Có khóa/mở khóa không?
- Có tải xuống/xuất file không?
- Có xem lịch sử thao tác không?
- Có import/upload không?
- Có phân quyền hiển thị button/function không?
- Có duyệt/từ chối/thu hồi/hủy nếu nghiệp vụ có trạng thái không?
- Có ghi log/audit cho thao tác thay đổi dữ liệu không?

## Mode B: Viết Use Case cho UI cấu hình/tích hợp hệ thống

Dùng khi màn hình phục vụ cấu hình hệ thống, cấu hình kết nối, cấu hình tham số, mapping dữ liệu, cấu hình lịch đồng bộ.

### Cấu trúc tên Use Case

```text
Quản lý cấu hình [Tên cấu hình/tích hợp]
```

hoặc:

```text
Cấu hình [Tên kết nối/tham số/mapping]
```

### Transaction thường gặp

| Nhóm function | Mẫu Transaction |
|---|---|
| Xem danh sách cấu hình | `[Tác nhân] xem danh sách cấu hình [Tên cấu hình]. Phần mềm hiển thị danh sách cấu hình [Tên cấu hình].` |
| Khởi tạo cấu hình | `[Tác nhân] khởi tạo cấu hình [Tên cấu hình]. Phần mềm kiểm tra thông tin cấu hình, nếu hợp lệ lưu cấu hình và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| Cập nhật cấu hình | `[Tác nhân] cập nhật cấu hình [Tên cấu hình]. Phần mềm kiểm tra thông tin thay đổi, nếu hợp lệ cập nhật cấu hình và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |
| Kiểm tra kết nối | `[Tác nhân] kiểm tra kết nối [Tên hệ thống/kết nối]. Phần mềm gửi yêu cầu kiểm tra kết nối và hiển thị kết quả kết nối thành công/thất bại.` |
| Kích hoạt/Vô hiệu hóa | `[Tác nhân] kích hoạt/vô hiệu hóa cấu hình [Tên cấu hình]. Phần mềm cập nhật trạng thái cấu hình và thông báo kết quả.` |
| Xem log đồng bộ | `[Tác nhân] xem log đồng bộ [Tên dữ liệu/kết nối]. Phần mềm hiển thị lịch sử xử lý và kết quả đồng bộ tương ứng.` |
| Đồng bộ thủ công | `[Tác nhân] thực hiện đồng bộ thủ công [Tên dữ liệu]. Phần mềm kiểm tra điều kiện, nếu hợp lệ thực hiện đồng bộ và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.` |

## Mode C: Viết Use Case tích hợp giữa phân hệ với phân hệ hoặc phân hệ với hệ thống

Dùng khi có trao đổi dữ liệu giữa phân hệ với phân hệ hoặc phân hệ với hệ thống. Theo dữ liệu mẫu, nhóm này dùng cho **UCs 2-3 Transaction về kết nối và trao đổi dữ liệu**.

### Nguyên tắc xác định chiều tích hợp

- **Hệ thống quản lý là hệ thống nguồn**: viết theo **luồng gửi dữ liệu**, tức hệ thống nguồn gửi dữ liệu đến hệ thống đích.
- **Hệ thống quản lý là hệ thống đích**: viết theo **luồng nhận dữ liệu**, tức hệ thống đích nhận dữ liệu từ hệ thống nguồn.
- Không tự thêm Transaction ngoài mẫu nếu người dùng chưa cung cấp thêm function/luồng xử lý.
- Giữ đúng vai trò hệ thống nguồn/hệ thống đích trong từng câu Transaction.

### Luồng gửi dữ liệu

Dùng khi **hệ thống quản lý là hệ thống nguồn** gửi dữ liệu đến hệ thống đích.

| STT | Mẫu Transaction chuẩn |
|---:|---|
| 1 | `[Tên hệ thống đích] gửi yêu cầu kết nối để khai thác thông tin [Tên dữ liệu] tới [Tên hệ thống nguồn]. [Tên hệ thống nguồn] xác thực yêu cầu, nếu hợp lệ thì cho phép kết nối, nếu không hợp lệ thì thông báo lại cho [Tên hệ thống đích].` |
| 2 | `[Tên hệ thống đích] gửi yêu cầu khai thác thông tin [Tên dữ liệu] đến [Tên hệ thống nguồn]. [Tên hệ thống nguồn] kiểm tra yêu cầu, nếu hợp lệ thì gửi dữ liệu [Tên dữ liệu] đến [Tên hệ thống đích], nếu không hợp lệ thì thông báo lỗi.` |
| 3 | `[Tên hệ thống đích] nhận dữ liệu [Tên dữ liệu] và gửi phản hồi cho [Tên hệ thống nguồn] (nếu có). [Tên hệ thống nguồn] ghi nhật ký xử lý.` |

### Luồng nhận dữ liệu

Dùng khi **hệ thống quản lý là hệ thống đích** nhận dữ liệu từ hệ thống nguồn.

| STT | Mẫu Transaction chuẩn |
|---:|---|
| 1 | `[Tên hệ thống nguồn] gửi yêu cầu kết nối tới [Tên hệ thống đích]. [Tên hệ thống đích] tiếp nhận yêu cầu và phản hồi kết quả kết nối thành công/thất bại cho [Tên hệ thống nguồn].` |
| 2 | `[Tên hệ thống nguồn] gửi dữ liệu [Tên loại dữ liệu] tới [Tên hệ thống đích]. [Tên hệ thống đích] tiếp nhận dữ liệu và ghi nhật ký xử lý.` |

### Checklist cho Use Case tích hợp

Kiểm tra đủ các ý sau:

- Đã xác định hệ thống quản lý đang là hệ thống nguồn hay hệ thống đích chưa?
- Nếu là luồng gửi dữ liệu, đã dùng đủ 3 Transaction mẫu chưa?
- Nếu là luồng nhận dữ liệu, đã dùng đúng 2 Transaction mẫu chưa?
- Có đúng tên hệ thống nguồn/hệ thống đích trong từng Transaction không?
- Có đúng tên dữ liệu/loại dữ liệu trao đổi không?
- Có Transaction kết nối không?
- Có Transaction khai thác/gửi dữ liệu không?
- Có phản hồi hoặc ghi nhật ký theo đúng mẫu không?
- Có Transaction nào bị tự thêm ngoài dữ liệu mẫu không? Nếu có, chỉ giữ khi người dùng cung cấp thêm nghiệp vụ cụ thể.

## Mode D: Viết Use Case cho phân hệ Trục tích hợp dữ liệu

Dùng khi hệ thống có phân hệ **Trục tích hợp dữ liệu**. Theo dữ liệu mẫu, nhóm này áp dụng cho các hệ thống có phân hệ Trục tích hợp và dùng cho **UCs 2-3 Transaction về kết nối và vận chuyển/forward dữ liệu, lưu log vận chuyển**.

### Cấu trúc Transaction chuẩn

Chỉ dùng các mẫu Transaction dưới đây làm cấu trúc mặc định, không tự tách thêm bước “tiếp nhận và kiểm tra” hoặc “chuyển tiếp” thành Transaction riêng nếu dữ liệu đầu vào chưa yêu cầu.

| STT | Mẫu Transaction chuẩn |
|---:|---|
| 1 | `[Tên hệ thống nguồn] gửi dữ liệu [Tên loại dữ liệu] tới Phần mềm Trục tích hợp dữ liệu. Phần mềm Trục tích hợp dữ liệu chuyển tiếp dữ liệu tới [Tên hệ thống đích].` |
| 2 | `[Tên hệ thống đích] tiếp nhận dữ liệu [Tên loại dữ liệu] và gửi phản hồi cho Phần mềm Trục tích hợp dữ liệu (nếu có). Phần mềm Trục tích hợp dữ liệu ghi nhật ký xử lý.` |

### Checklist cho Trục tích hợp dữ liệu

- Có hệ thống nguồn không?
- Có hệ thống đích không?
- Có loại dữ liệu vận chuyển không?
- Transaction 1 có đúng cấu trúc: nguồn gửi dữ liệu tới Trục, Trục chuyển tiếp tới đích không?
- Transaction 2 có đúng cấu trúc: đích tiếp nhận, phản hồi nếu có, Trục ghi nhật ký xử lý không?
- Có bị tách sai thành quá nhiều Transaction ngoài mẫu không?
- Có dùng đúng cụm “Phần mềm Trục tích hợp dữ liệu” không?
- Có giữ đúng nội dung “gửi phản hồi ... (nếu có)” không?
- Có giữ đúng nội dung “ghi nhật ký xử lý” không?

## Mode E: Review/gap-check Use Case và Transaction

Dùng khi người dùng đưa danh sách Use Case/Transaction hiện có và yêu cầu rà soát.

### Quy trình review

1. Chuẩn hóa danh sách Use Case và Transaction thành bảng.
2. Kiểm tra trùng tên Use Case.
3. Kiểm tra trùng nội dung Transaction hoặc Transaction quá giống nhau.
4. Kiểm tra mỗi function màn hình đã có Transaction riêng chưa.
5. Kiểm tra thiếu function phổ biến theo nhóm UI/tích hợp.
6. Kiểm tra chiều tích hợp nguồn/đích có đúng không.
7. Kiểm tra câu mô tả có đủ tác nhân, hành động, phần mềm xử lý và kết quả không.
8. Đề xuất bản sửa cụ thể.

### Bảng review bắt buộc

Khi review, xuất bảng:

| STT | Use Case | Transaction hiện tại | Vấn đề | Mức độ | Gợi ý chỉnh sửa |
|---:|---|---|---|---|---|

Mức độ gồm:

- **Critical**: trùng, sai chiều tích hợp, thiếu Transaction chính, hoặc mô tả có thể dẫn đến hiểu sai nghiệp vụ.
- **Major**: thiếu tác nhân, thiếu phản hồi, thiếu xử lý lỗi, thiếu log, hoặc gộp nhiều function vào một Transaction.
- **Minor**: câu chữ chưa đồng nhất, tên chưa đúng chuẩn, thiếu chi tiết nhỏ.
- **Suggestion**: đề xuất tối ưu để tài liệu dễ đọc hoặc dễ bảo trì hơn.

### Checklist review nhanh

- Use Case có đúng cấu trúc tên không?
- Transaction có bắt đầu bằng tác nhân/hệ thống thực hiện không?
- Transaction có nêu phần mềm/hệ thống xử lý không?
- Transaction có kết quả đầu ra không?
- Có kiểm tra hợp lệ/thông báo lỗi khi thao tác thay đổi dữ liệu không?
- Có Transaction cho từng function không?
- Có Transaction bị trùng không?
- Có Use Case bị trùng không?
- Có thiếu log/audit cho thao tác quan trọng không?
- Có sai chiều nguồn/đích trong tích hợp không?

## Format output mặc định

Khi tạo mới, dùng bảng:

| STT | Nhóm Use Case | Tên Use Case | Tác nhân | Transaction | Ghi chú |
|---:|---|---|---|---|---|

Khi có nhiều Use Case, nhóm theo:

1. UI quản lý nghiệp vụ;
2. UI cấu hình/tích hợp;
3. Tích hợp hệ thống;
4. Trục tích hợp dữ liệu;
5. Nhóm khác.

## Mẫu câu chuẩn

### UI quản lý nghiệp vụ

```text
[Tác nhân] xem danh sách [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm hiển thị danh sách [Tên nghiệp vụ/dữ liệu quản lý].
```

```text
[Tác nhân] khởi tạo [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ thực hiện lưu thông tin và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.
```

```text
[Tác nhân] cập nhật thông tin [Tên nghiệp vụ/dữ liệu quản lý]. Phần mềm kiểm tra, nếu hợp lệ xử lý yêu cầu cập nhật dữ liệu và thông báo kết quả, nếu không hợp lệ thông báo lỗi cụ thể.
```

### Tích hợp hệ thống

```text
[Tên hệ thống đích] gửi yêu cầu kết nối để khai thác thông tin [Tên dữ liệu] tới [Tên hệ thống nguồn]. [Tên hệ thống nguồn] xác thực yêu cầu, nếu hợp lệ thì cho phép kết nối, nếu không hợp lệ thì thông báo lại cho [Tên hệ thống đích].
```

```text
[Tên hệ thống nguồn] gửi dữ liệu [Tên loại dữ liệu] tới [Tên hệ thống đích]. [Tên hệ thống đích] tiếp nhận dữ liệu và ghi nhật ký xử lý.
```

### Trục tích hợp dữ liệu

```text
[Tên hệ thống nguồn] gửi dữ liệu [Tên loại dữ liệu] tới Phần mềm Trục tích hợp dữ liệu. Phần mềm Trục tích hợp dữ liệu chuyển tiếp dữ liệu tới [Tên hệ thống đích].
```

```text
[Tên hệ thống đích] tiếp nhận dữ liệu [Tên loại dữ liệu] và gửi phản hồi cho Phần mềm Trục tích hợp dữ liệu nếu có. Phần mềm Trục tích hợp dữ liệu ghi nhật ký xử lý.
```

## Quy tắc đặt tên và chống trùng

### Đặt tên Use Case

Tên Use Case nên ngắn gọn, bắt đầu bằng động từ hoặc nhóm hành động chính:

- `Quản lý [Tên dữ liệu/nghiệp vụ]`
- `Cấu hình [Tên cấu hình]`
- `Khai thác thông tin [Tên dữ liệu] từ [Tên hệ thống nguồn]`
- `Tiếp nhận dữ liệu [Tên dữ liệu] từ [Tên hệ thống nguồn]`
- `Chuyển tiếp dữ liệu [Tên dữ liệu] qua Trục tích hợp dữ liệu`

### Chống trùng

Trước khi trả lời, tự rà soát:

- hai Use Case có cùng tên không;
- hai Use Case khác tên nhưng cùng mục tiêu không;
- hai Transaction khác câu chữ nhưng cùng hành động không;
- Transaction có bị lặp giữa UI và tích hợp không;
- Transaction có đang mô tả function quá nhỏ không cần tách hoặc quá lớn cần tách không.

Nếu phát hiện trùng, không xóa âm thầm. Hãy ghi rõ:

```text
Phát hiện trùng: [UC/Transaction A] và [UC/Transaction B].
Khuyến nghị: giữ [tên chuẩn], gộp/chuyển [nội dung] thành [tên mới].
```

## Gợi ý khi thiếu thông tin

Nếu thiếu tác nhân:

```text
Tạm giả định tác nhân là [Quản trị hệ thống/Người dùng nghiệp vụ/Hệ thống nguồn/Hệ thống đích]. Cần xác nhận lại.
```

Nếu thiếu hệ thống nguồn/đích:

```text
Cần xác định hệ thống nguồn và hệ thống đích để viết đúng chiều tích hợp.
```

Nếu thiếu danh sách function:

```text
Tạm đề xuất bộ function chuẩn gồm xem danh sách, thêm mới, tìm kiếm, lọc, cập nhật, xóa, xem chi tiết, khóa/mở khóa, tải xuống và xem lịch sử thao tác. Cần đối chiếu với màn hình thực tế.
```

## Definition of Done

Một đầu ra Use Case/Transaction được coi là đạt khi:

- tên Use Case rõ ràng, không trùng;
- mỗi Transaction mô tả đúng một hành động/function chính;
- Transaction có tác nhân, hành động, phần mềm/hệ thống xử lý và kết quả;
- các thao tác tạo/cập nhật/xóa có kiểm tra hợp lệ và thông báo lỗi;
- các luồng tích hợp có đúng hệ thống nguồn, hệ thống đích, dữ liệu trao đổi, phản hồi và log;
- các Use Case Trục tích hợp có tiếp nhận, chuyển tiếp và ghi nhật ký vận chuyển;
- danh sách đã được rà soát chống trùng;
- các giả định/câu hỏi mở được liệt kê rõ ràng.
