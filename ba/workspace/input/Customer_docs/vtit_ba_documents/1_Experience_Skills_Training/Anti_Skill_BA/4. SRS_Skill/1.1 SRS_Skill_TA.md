## skill-ba-srs-writer



## Description: write, refine, review, or expand software requirement specification documents for business analysts. use when the user needs an srs, brd-to-srs conversion, feature specification, screen specification, business flow, api/db specification, validation matrix, edge-case checklist, or review checklist for a system, module, feature, or change request. especially use for producing testable, non-ambiguous, logically ordered requirements that cover ui, business rules, exceptions, integrations, data governance, and reusable scenarios.

# BA SRS Writer

Write SRS documents for Business Analysts in a way that is **clear, testable, structured, and hard to miss cases**.

This skill is optimized for AI-assisted BA work: turning business input, meeting notes, feature ideas, existing docs, screenshots, API notes, or function lists into a complete SRS draft or a gap-reviewed SRS.

## Core objective

Produce SRS content that:

* follows a **top-down flow**: context → scope → functional overview → detailed design → business flow → technical handling → data governance → checklist/review;
* is written in a way that **QA can test**, **Dev can build**, and **UAT can verify**;
* makes assumptions explicit and avoids vague wording;
* covers normal flow, alternate flow, exception flow, empty data, permission cases, and data source ownership;
* supports reuse by separating **common scenarios** from feature-specific logic.

## Non-negotiable writing principles

Always apply these rules:

1. Write requirements so they can be verified.
2. Replace vague language with measurable or observable behavior.
3. Order sections by actual user/system flow.
4. State data source clearly: API, DB, service, cache, file, third-party system.
5. Specify validation, error handling, empty-state handling, and permission handling.
6. Distinguish fact, assumption, open question, and recommendation.
7. Reuse common scenarios in one place and reference them instead of repeating.
8. Keep terminology consistent throughout the document.
9. Name actors, triggers, preconditions, postconditions, and outputs explicitly.
10. When information is missing, do not silently guess critical business logic. Mark it as an open item.

## Input handling

Accept any of these as source material:

* project brief;
* PYC/change request;
* BRD, URD, meeting notes;
* screenshots/mockups/Figma descriptions;
* API specs;
* database notes;
* function list;
* legacy SRS or current-state documentation;
* checklist or review comments.

When input is incomplete, do the following in order:

1. Extract all explicit facts.
2. Infer only low-risk structural details.
3. Mark high-risk missing details under **Open Questions / Assumptions**.
4. Continue drafting with best-effort structure instead of stopping.

## Standard output modes

Choose the closest mode to the user’s request.

### Mode A: Create a full SRS

Use when the user asks to write a new SRS.

### Mode B: Expand a partial SRS

Use when the user already has headings, flow, wireframes, or rough notes.

### Mode C: Review and gap-check an SRS

Use when the user provides an existing draft and wants missing cases, ambiguity detection, or quality review.

### Mode D: Write one feature/module spec

Use when the request is for a single feature, single screen, or one change request.

## Required SRS skeleton

Use this structure unless the user requests another company template.

### 1\. Document overview

* Document purpose
* Scope
* Stakeholders
* Definitions / terminology
* References / source documents
* Assumptions / dependencies

### 2\. Solution overview

#### 2.1 Functional overview

* Brief description of feature/system purpose
* Related modules / external systems
* High-level actor interaction

#### 2.2 Scope detail

* In scope
* Out of scope
* Impacted modules
* Reused/common scenarios

### 3\. Detailed design

For each function or sub-function, write the following.

#### 3.1 General information

Use a table or bullet form with:

* Function code
* Function name
* Purpose
* Target users / roles
* Access path
* Trigger
* Preconditions
* Postconditions
* Related business rules
* Related APIs / DB tables / services

#### 3.2 Screen overview

Describe:

* screen purpose;
* layout zones;
* major components;
* default state;
* loaded state;
* empty state;
* error state;
* permission-based visibility if applicable.

#### 3.3 Detailed screen component specification

For each field/control, specify at minimum:

* No.
* Component name
* Data type
* Input / Output
* Required or optional
* Default value
* Editable / read-only condition
* Validation rules
* Format / length / allowed values
* Placeholder / label / tooltip
* Data source
* Display logic
* Notes

Always include handling for:

* null / empty data;
* overly long data;
* invalid format;
* duplicate data;
* inactive/deleted records;
* role-based disable/hide behavior.

#### 3.4 Business flow

Provide:

* trigger;
* actor;
* main flow;
* alternate flow;
* exception flow;
* completion result.

Write detailed flow in this pattern:

* Initialization
* Execution
* System validations
* Error/exception handling
* Success result
* Failure result
* Audit/logging if applicable

#### 3.5 Technical handling

##### API

For each relevant API, specify:

* Endpoint
* Method
* Purpose
* Caller / consumer
* Auth requirement
* Request parameters
* Header requirements
* Request body
* Success response
* Error response
* Retry / timeout / idempotency note if applicable
* Mapping to UI/business action

##### Database

For each impacted table, specify:

* Table name
* Purpose
* Read / write action
* Insert / update / delete logic
* Key fields
* Validation / constraints
* Soft delete / hard delete rule
* Audit columns
* Data lineage / owner if known

### 4\. Common reusable scenarios

Separate shared scenarios from screen-specific logic, such as:

* soft delete;
* approval action;
* export file;
* import file;
* pagination/filter/sort/search;
* attachment upload/download;
* notification sending;
* status transition;
* session timeout;
* role/permission enforcement.

For each common scenario, specify:

* objective;
* scope of reuse;
* actor;
* pre/post conditions;
* business rules;
* technical dependencies;
* reusable error handling.

### 5\. Data governance and compliance

When relevant, include:

* data field inventory;
* table-field mapping;
* data type;
* owner;
* source of truth;
* access classification;
* retention or deletion behavior;
* quality check rules.

### 6\. Non-functional requirements

Write only if applicable, but be concrete:

* performance;
* security;
* availability;
* usability;
* auditability;
* localization;
* browser/device support.

Avoid generic phrases such as “fast”, “secure”, or “user friendly”. Make them testable.

### 7\. Open questions / assumptions / risks

Always add this section if any business rule is missing or inferred.

## Business coverage checklist

Before finishing, self-check that the draft covers:

* objective and scope;
* actors and permissions;
* entry point/access path;
* preconditions/postconditions;
* all fields/components;
* validations;
* default/empty/error/loading states;
* business rules;
* main/alternate/exception flows;
* success/failure outcomes;
* API handling;
* DB handling;
* impacted modules/integrations;
* logging/audit if relevant;
* reusable common scenarios;
* open questions;
* non-functional constraints if relevant.

## High-risk omission checklist

Use this aggressively when the feature involves authentication, state, or user actions.

### Authentication and session

Check for:

* login required or not;
* supported login types: local, SSO, email, linked account;
* remember me behavior;
* forgot password behavior;
* captcha policy;
* failed login threshold and temporary lock;
* session storage and lifetime;
* token refresh flow;
* concurrent login policy;
* behavior when role changes during active session;
* logout handling;
* login/logout logging.

### Password management

Check for:

* current password validation;
* new password policy;
* confirmation mismatch;
* forced logout after password change or not;
* verification code/OTP requirement;
* impact on other active sessions.

### Common UI behavior

Check for:

* search/filter/sort/reset;
* pagination;
* export/import;
* bulk actions;
* attachment rules;
* confirmation popup;
* empty list behavior;
* permission-based button visibility;
* loading skeleton/spinner;
* duplicate submission prevention.

### Data handling

Check for:

* source API/DB;
* stale data/reload behavior;
* soft delete vs hard delete;
* uniqueness rules;
* reference data lookup;
* length/format/range validations;
* null handling;
* data masking if sensitive.

## Writing style rules

* Use concise, operational language.
* Prefer bullets and tables for dense specs.
* Use bold only for important rules or warnings.
* Keep bullets at the same level grammatically parallel.
* Use one term for one concept.
* Avoid mixing business requirement and implementation guess in the same sentence.

## Wording patterns

Use these sentence patterns when drafting.

### Function purpose

* Purpose: \[short business outcome].
* Target users: \[roles allowed to access].
* Access path: \[menu / URL / navigation path].
* Preconditions: \[conditions that must already be true].
* Postconditions: \[state after completion].

### Business flow

* The system checks \[condition].
* If \[condition] is met, the system \[action].
* If \[condition] is not met, the system \[error handling].
* The user performs \[action].
* The system returns \[result].

### Validation

* The field is required when \[condition].
* The value must match \[format/range/rule].
* If invalid, the system displays \[message/behavior].
* The system does not allow submission when \[condition].

### API handling

* The system calls \[method] \[endpoint] to \[purpose].
* Request parameters include \[fields].
* On success (200/201), the system \[behavior].
* On business error (4xx), the system \[behavior].
* On system error (5xx/timeout), the system \[behavior].

### Database handling

* The system reads data from \[table].
* The system updates \[field] in \[table] when \[condition].
* The record is soft-deleted by setting \[field/value].
* Audit information includes \[created\_by, created\_at, updated\_by, updated\_at].

## Output formatting rules for AI responses

When generating the SRS in chat, structure the answer in this order:

1. Summary of understanding
2. Scope and assumptions
3. SRS content
4. Open questions
5. Gap/risk checklist

If the user asks for a document artifact, generate in Markdown with clear headings and tables.

## Review mode behavior

When reviewing an existing SRS:

1. Check completeness.
2. Check consistency of terms and rules.
3. Check clarity and testability.
4. Check traceability to source request.
5. Check UI-business-API-DB alignment.
6. List missing cases explicitly.
7. Suggest exact rewritten text, not only criticism.

Use this review classification:

* Critical: blocks dev/QA/UAT or risks wrong implementation.
* Major: likely to cause misunderstanding or missed cases.
* Minor: formatting, wording, or maintainability issue.

## Definition of done

A draft is only considered complete when:

* the requirement is testable;
* business flow is logically ordered;
* fields and validations are specified;
* success, failure, and exception cases are covered;
* API/DB touchpoints are identified when relevant;
* data source is stated;
* reusable scenarios are separated;
* open questions are listed instead of hidden.

## Preferred response behavior

* Do the drafting work proactively.
* Do not stop just because some inputs are missing.
* Draft the best complete structure possible with clearly marked assumptions.
* When a request is too broad, break it into modules/features and continue.
* When the user provides a checklist, convert it into embedded quality gates inside the SRS.
