## 1. CẤU TRÚC BẮT BUỘC CỦA MỖI CHỨC NĂNG

Mỗi chức năng (màn hình) phải viết đủ **6 mục** theo thứ tự sau:

```
3.X.Y   Tên chức năng
3.X.Y.1 Thông tin chung chức năng
3.X.Y.2 Tổng quan màn hình
3.X.Y.3 Mô tả chi tiết các thành phần màn hình
3.X.Y.4 Luồng nghiệp vụ chi tiết
3.X.Y.5 Xử lý phần kỹ thuật (API & DB)
3.X.Y.6 Kịch bản dùng chung và Tuân thủ
```

## 2. CHI TIẾT TỪNG MỤC

### 2.1 Thông tin chung chức năng (3.X.Y.1)

Dùng bảng 2 cột. Bắt buộc có đủ các dòng:

| Trường | Ghi chú |
|---|---|
| Mã chức năng | Đặt theo quy ước: `FUNC_[MODULE]_[NNN]` |
| Tên chức năng | Tên đầy đủ của chức năng |
| Mục đích | 1-2 câu ngắn gọn mô tả mục đích nghiệp vụ |
| Người dùng mục tiêu | Liệt kê role + điều kiện quyền truy cập |
| Đường dẫn truy cập | Mô tả luồng điều hướng cụ thể từ menu/màn hình khác |
| Trigger | Sự kiện nào kích hoạt màn hình/chức năng này |
| Tiền điều kiện | Điều kiện phải đúng TRƯỚC khi thực hiện |
| Hậu điều kiện | Trạng thái hệ thống SAU khi hoàn tất |

---

### 2.2 Tổng quan màn hình (3.X.Y.2)

- Mô tả **bố cục tổng thể** (số vùng, tên từng vùng).
- Quy tắc chung áp dụng cho toàn màn hình (không lặp lại ở từng trường).

**Quy tắc chung BẮT BUỘC phải có trên màn hình Danh sách:**
```
- Quy tắc dữ liệu trống: null/undefined/"" → hiển thị dấu "-" căn giữa ô.
- Quy tắc cắt chuỗi (Text Overflow): 1 dòng cố định, cắt "...", tooltip khi hover.
```

**Quy tắc chung BẮT BUỘC phải có trên màn hình Form (Thêm/Sửa):**
```
- Trường (*) là bắt buộc. Validate khi nhấn Lưu.
- Lỗi validate hiển thị inline ngay dưới từng trường.
- Popup cảnh báo khi Huỷ nếu đã có dữ liệu nhập.
```

### 2.3 Mô tả chi tiết các thành phần (3.X.Y.3)

#### Cấu trúc bảng chuẩn (8 cột):

| STT | Tên thành phần | Kiểu DL | In/Out | Bắt buộc | Mô tả chi tiết & Quy tắc nghiệp vụ | Mapping API | Mapping DB |
|---|---|---|---|---|---|---|---|

#### Quy tắc nhóm hàng (Group Row):
- Dùng chữ La Mã (I, II, III, IV...) để nhóm theo vùng màn hình.
- Hàng nhóm: in đậm, nền màu xanh nhạt.
- Nút Header dùng ký hiệu H1, H2, H3...

#### Kiểu dữ liệu chuẩn cần dùng:

| Kiểu | Dùng khi |
|---|---|
| `Text` | Ô nhập chữ tự do |
| `Text (Read-only)` | Hiển thị, không cho nhập tay |
| `Password` | Ô nhập mật khẩu |
| `Number` | Chỉ nhập số |
| `Date Picker` | Chọn ngày tháng |
| `Dropdown (DDL)` | Chọn từ danh sách |
| `Radio Group` | Chọn 1 trong nhiều giá trị |
| `Toggle Switch` | Bật/tắt trạng thái |
| `Button` | Nút thao tác thường |
| `Button (Primary)` | Nút hành động chính |
| `Icon Menu` | Nút icon mở dropdown |
| `Badge` | Hiển thị trạng thái dạng nhãn màu |
| `Paging` | Điều hướng phân trang |

#### Checklist BẮT BUỘC cho mỗi trường Input:
```
☐ Placeholder text là gì?
☐ Định dạng dữ liệu hợp lệ? (chữ/số/ký tự đặc biệt/có dấu/không dấu)
☐ Độ dài Min/Max?
☐ Giá trị mặc định (Default)?
☐ Có validate unique không? Nếu có: gọi API khi nào (blur/submit)?
☐ Thông báo lỗi inline là gì?
☐ Trường này editable hay read-only theo điều kiện nào?
☐ Nguồn dữ liệu (nếu là Dropdown): fix cứng hay lấy từ danh mục nào/phân hệ nào?
```

#### Checklist BẮT BUỘC cho mỗi trường Output (hiển thị trong bảng/form):
```
☐ Mapping với field API response nào? (Res: fieldName)
☐ Mapping với bảng/cột DB nào?
☐ Nếu dữ liệu null → hiển thị gì? (mặc định "-")
☐ Nếu dữ liệu quá dài → xử lý thế nào? (mặc định cắt + tooltip)
☐ Có định dạng hiển thị đặc biệt không? (ngày tháng, số điện thoại, tiền tệ...)
☐ Có gắn link redirect không? Nếu có → redirect đến đâu?
☐ Có màu sắc/badge theo trạng thái không?
```

### 2.4 Luồng nghiệp vụ chi tiết (3.X.Y.4)

Mỗi kịch bản dùng bảng 2 cột (Bước | Mô tả). **BẮT BUỘC** có đủ:

```
A. Kịch bản khởi tạo/Load màn hình
B. Kịch bản luồng chính (Happy Path)
C. Kịch bản huỷ bỏ / thoát
D. (Nếu có) Các kịch bản ngoại lệ đặc thù
```

**Trong mỗi kịch bản BẮT BUỘC mô tả:**

| Điểm | Nội dung cần có |
|---|---|
| **Loading** | Thành phần nào hiển thị loading? Skeleton/Spinner/Overlay? Chặn thao tác gì? |
| **Validate** | Validate frontend trước khi gọi API không? Realtime (blur) hay chỉ khi submit? |
| **Gọi API** | Endpoint nào? Method gì? |
| **Xử lý 403** | Không có quyền → xử lý thế nào? |
| **Xử lý 4xx** | Lỗi nghiệp vụ → Toast/inline gì? |
| **Xử lý 5xx/Timeout** | Lỗi hệ thống → thông báo gì? Giữ dữ liệu đã nhập không? |
| **Thành công** | Toast gì? Điều hướng đến đâu? Reload gì? Ghi log không? |
| **Chống double-click** | Nút bị disable trong khi chờ API không? |

---

### 2.5 Xử lý phần kỹ thuật (3.X.Y.5)

**Với mỗi API**, dùng bảng 2 cột gồm: Endpoint, Mục đích, Tham số Request, Response thành công, Response lỗi.

**Danh sách API hay gặp theo loại màn hình:**

| Loại màn hình | API thường gặp |
|---|---|
| Danh sách | GET list (filter+sort+page), PATCH status, GET export, GET check-unique |
| Form Thêm mới | GET danh mục dropdown, GET generate-username, GET check-unique, POST create |
| Form Sửa | GET detail, PUT/PATCH update, GET check-unique |
| Xóa | DELETE hoặc PATCH (soft delete) |
| Lịch sử | GET history / audit-log |

**Với mỗi bảng DB**, ghi rõ: Tên bảng, Mục đích, Hành động (READ/INSERT/UPDATE), Ghi chú.

**Các trường audit BẮT BUỘC khi INSERT/UPDATE:**
```
created_at = NOW(), created_by = [ID actor]
updated_at = NOW(), updated_by = [ID actor]
is_deleted = false  (soft delete)
```

### 2.6 Kịch bản dùng chung (3.X.Y.6)

Viết một lần, tham chiếu lại – không lặp. Các kịch bản common phổ biến:

| Kịch bản | Nội dung cần có |
|---|---|
| Export Excel | Tên file, giới hạn bản ghi, xử lý vượt ngưỡng (chặn + hướng dẫn thu hẹp lọc) |
| Phân trang | Reset về trang 1 khi filter/đổi pageSize |
| Loading UX | Skeleton hoặc Overlay Spinner – không làm trắng/giật trang |
| Chống submit trùng | Disable nút ngay khi click, enable lại sau khi nhận response |
| Soft Delete | Luôn WHERE is_deleted = false khi query |
| Audit Log | Ghi mọi hành động: tạo/sửa/xóa/đổi trạng thái/export |
| Gửi email async | Đẩy vào queue, không block luồng chính, ghi log nếu lỗi gửi |

## 3. QUY TẮC VALIDATE THEO TỪNG LOẠI DỮ LIỆU

| Loại dữ liệu | Quy tắc chuẩn |
|---|---|
| **Text tự do** | Có dấu, khoảng trắng. Max 255 (tên), 2000 (mô tả). Trim 2 đầu. |
| **Mã định danh (Code)** | Không dấu, không khoảng trắng, không ký tự đặc biệt. Chỉ a-z/A-Z/0-9. Max 50. Unique. |
| **Mã người dùng (auto-gen)** | Chỉ số. Tự sinh mặc định, cho phép sửa. Unique. |
| **Số điện thoại** | Chỉ số, bắt đầu 0. Min 10, Max 11. Lưu dạng string. Export Excel format Text. |
| **Email** | Chuẩn name@domain.ext. Max 255. Unique (case-insensitive). |
| **Mật khẩu** | Min 8. Có chữ thường + hoa + số + ký tự đặc biệt. Không khoảng trắng. Không chứa username/fullname. Hash trước khi lưu. |
| **Ngày tháng** | Format DD/MM/YYYY. Hỗ trợ nhập tay + picker. Validate ngày hợp lệ. Quy định thêm: không tương lai / >= 18 tuổi... tuỳ trường. |
| **Mã số thuế** | Kiểu 1: 10 số. Kiểu 2: 10 số + "-" + 3 số (= 14 ký tự). Unique. |
| **Mã chứng khoán** | Chữ cái không dấu hoặc số, hoặc kết hợp. Max 10. Unique. |
| **Số CCCD** | Đúng 12 chữ số. |
| **Địa chỉ IP** | IPv4: 4 octet 0-255 cách nhau ".". Max 45 (tương thích IPv6). |
| **Dropdown từ danh mục** | Ghi rõ nguồn: tên danh mục + tên phân hệ. Gọi API khi load. Loading khi đang tải. Disable nếu API lỗi. |
| **Dropdown fix cứng** | Liệt kê tất cả giá trị. Ghi rõ Default. |

## 4. QUY TẮC SINH TÊN TÀI KHOẢN (AUTO-GEN USERNAME)

```
Rule: [Tên] + [Chữ cái đầu Họ] + [Chữ cái đầu Tên đệm 1] + [Chữ cái đầu Tên đệm 2 (nếu có)]
Chuẩn hoá: Bỏ dấu tiếng Việt → Chữ thường → Bỏ khoảng trắng
Xử lý trùng: Thêm số tự tăng vào cuối (1, 2, 3...)

Ví dụ:
  Nguyễn Văn An    → an + n + v = "annv"
  Trần Thị Bích Ngọc → ngoc + t + t + b = "ngoctb"
  "annv" đã tồn tại → thử "annv1" → "annv2"...

Trigger: Gọi API GET /api/v1/users/generate-username?fullName={fullName} khi blur khỏi trường Tên người dùng
```

## 6. CHECKLIST TRƯỚC KHI HOÀN TẤT

### Màn hình Danh sách:
```
☐ Đủ 4 vùng: Header actions, Filter, Grid, Pagination
☐ Mỗi trường bảng: Mapping API (Res) + Mapping DB
☐ Quy tắc dữ liệu trống (-) và Text Overflow (...)
☐ Toggle Switch: quy tắc ON/OFF + xử lý không có quyền (Toast đỏ)
☐ Nút Export: tên file + giới hạn bản ghi + xử lý vượt ngưỡng (chặn + hướng dẫn)
☐ Dropdown thao tác: ẩn/hiện theo quyền
☐ Loading: Get list / Toggle / Export
☐ Empty state khi không có dữ liệu
☐ Xử lý lỗi: 403, 5xx/Network
☐ Validate bộ lọc (độ dài, định dạng) trước khi gọi API
```

### Màn hình Form Thêm mới:
```
☐ Mô tả đủ các vùng trên form
☐ Mỗi trường: placeholder, định dạng, min/max, default, validate, thông báo lỗi inline
☐ Dropdown: nguồn dữ liệu + API lấy danh mục + xử lý lỗi load danh mục
☐ Trường auto-gen (Read-only): quy tắc sinh + trigger cập nhật
☐ Validate unique: gọi API khi blur, thông báo lỗi inline cụ thể
☐ Nút Lưu: Loading + chống double-click + xử lý 201/4xx/5xx
☐ Nút Huỷ: Popup xác nhận nếu đã có dữ liệu
☐ Radio thay đổi loại: popup cảnh báo mất dữ liệu
☐ Loading khi load danh mục ban đầu
☐ Hành động sau lưu thành công: Toast, redirect, log, email...
```

## 7. MẪU CÂU CHUẨN THƯỜNG DÙNG

```markdown
# Validate trường nhập
Định dạng: Chỉ nhận chữ cái không dấu (a-z, A-Z) và số (0-9).
Không nhận khoảng trắng, ký tự đặc biệt.
Độ dài: Min [X] ký tự, Max [Y] ký tự.
Validate unique: Kiểm tra trùng khi blur. Nếu trùng → lỗi inline:
"[Tên trường] đã tồn tại trong hệ thống".

# Dropdown từ danh mục
Nguồn dữ liệu: Lấy động từ danh mục "[Tên danh mục]"
thuộc Phân hệ "[Tên phân hệ]".
Gọi API khi load form. Hiển thị Loading trong Dropdown khi đang tải.
Nếu API danh mục lỗi → disable Dropdown + hiển thị thông báo lỗi.

# Toggle không có quyền
Toggle Switch vẫn luôn hiển thị trạng thái ON/OFF hiện tại.
Nếu cố thao tác: hệ thống chặn, trạng thái không đổi,
hiển thị Toast đỏ: "Bạn không có quyền thực hiện chức năng này".

# Export vượt ngưỡng
Nếu totalItems > [N] bản ghi: Hệ thống không thực hiện Export.
Toast đỏ: "Số lượng dữ liệu xuất quá lớn (tối đa [N] bản ghi/lần).
Vui lòng thu hẹp phạm vi tìm kiếm để thực hiện kết xuất".

# Loading gọi API
Hiển thị [Skeleton/Overlay Spinner/Spinner trên nút].
[Tên nút] bị disable tạm thời chống double-click.
Gọi API [METHOD] [endpoint].

# Lỗi hệ thống
Tắt Loading. Toast đỏ: "Đã xảy ra lỗi hệ thống. Vui lòng thử lại sau."
Giữ nguyên dữ liệu đã nhập trên form.

# Thành công
API trả HTTP [201/200]. Tắt Loading.
Toast xanh: "[Tên hành động] thành công".
Ghi Audit Log (actor, action, target_id, timestamp, ip_address).
Chuyển về [Màn hình] / thực hiện [Hành động tiếp theo].
```

## 8. CÂU HỎI MỞ – NHỮNG ĐIỂM LUÔN CẦN HỎI KHI VIẾT SRS

```
☐ Dropdown lấy từ danh mục nào / phân hệ nào?

☐ Xóa theo kiểu soft delete hay hard delete?
☐ Có gửi thông báo (email/notification) sau thao tác không?
☐ Mẫu email/template: fix cứng hay cấu hình động? Ai quản lý?
☐ Giới hạn xuất Excel bao nhiêu bản ghi?
☐ Toggle trạng thái: có force logout user không khi bị khóa?
☐ Validate unique: gọi khi blur hay chỉ khi submit?
☐ Trường auto-gen: trigger cập nhật khi nào? Có cho sửa không?
☐ Xử lý khi API danh mục (dropdown) load lỗi: disable hay vẫn cho nhập?
```
